# AI活用プロダクト化メモ — 友人協議MTG用

> 友人とのAI活用協議MTGで使用するメモです。特定技能2号移行ナビの構築プロセスを題材に、「非エンジニアが専門知識をプロダクト化し、営業資料まで展開するモデル」について議論するための材料をまとめています。

---

## 背景

このリポジトリ（tokutei-navi-01）は、外食業の特定技能2号移行という専門領域の知識を、Lovable（AIアプリビルダー）とGitHub、そしてClaude（AIアシスタント）を組み合わせて、診断ツール・正本管理・営業資料化まで一気通貫で構築した事例です。

エンジニアがゼロからコードを書くのではなく、**業務知識を持つ人がAIツールを使ってプロダクトと商談資料の両方を作る**という流れそのものが、友人との協議における中心テーマです。

---

## 構築モデルの全体像

```
① Lovable でUI化
   HTML原案 → React/TypeScriptアプリへ変換（チャット指示ベース）
        ↓
② GitHub で正本管理
   Lovableが自動pushする形で、コードの唯一の正本をリポジトリ側に固定
        ↓
③ README でプロダクト定義
   何のツールか、誰向けか、何を入力し何を出力するかを明文化
        ↓
④ runtime_rules でロジック保護
   AIによる意図しないロジック改変・SEOタグ破壊を防ぐルールを明文化
        ↓
⑤ CONTRIBUTING でAI開発フロー管理
   Claude→GPT→Lovable→Publishという役割分担と手順を固定化
        ↓
⑥ docs/ で営業資料・PoC資料化
   同じ正本から、1枚資料・スライド構成・ヒアリングシート・PoC設計書を生成
```

各層が「誰が」「何のために」触るファイルかを分離している点が、この構築モデルの特徴です。

---

## 各層の役割整理

| 層 | 役割 | 主な担い手 |
|---|---|---|
| Lovable（UI化） | チャット指示によるReactアプリの生成・修正 | 非エンジニア（プロンプト作成） |
| GitHub（正本管理） | コードの唯一の正本を保持し、変更履歴を追跡 | 自動push＋人による確認 |
| README（プロダクト定義） | プロダクトの目的・対象・入出力を第三者にも分かる形で固定 | 人（Claude支援） |
| runtime_rules（ロジック保護） | AIがロジック・SEOタグを誤って書き換えないための凍結ルール | 人（過去のトラブルから学習・明文化） |
| CONTRIBUTING（AI開発フロー） | Claude/GPT/Lovableの役割分担と修正手順の標準化 | 人（運用ルールとして明文化） |
| docs/（営業・PoC資料化） | 同じ正本情報から、面談・PoC・法務向け資料を展開 | 人（Claude支援） |

---

## このモデルが示していること

- **非エンジニアでも、専門知識（特定技能制度の実務知識）を診断ロジックとして構造化し、動くプロダクトに変換できる**。
- コードを直接書く代わりに、「何を守るか（runtime_rules）」「どう依頼するか（lovable_prompt, CONTRIBUTING）」を明文化することで、AIによる誤改変のリスクをコントロールできる。
- プロダクトのコード（src/）と、営業・PoC資料（docs/）を同一リポジトリ内で分離しつつ連携させることで、**「作る」と「売る・検証する」を同じ正本から一貫して展開できる**。
- 1つのツールを単体で終わらせず、PreCheck・インテグリティ診断という複数プロダクトを接続したファネル設計にすることで、単発ツールから「意思決定プロセスの提供」へと価値を拡張している。
- 営業資料・ヒアリングシート・PoC設計書・免責事項までをリポジトリ内のMarkdownとして管理することで、口頭伝承や個人のPC内資料に依存しない、**再現可能な営業オペレーション**になっている。

## 友人との協議論点

1. **このモデルは他の専門領域にも横展開できるか**
   特定技能制度に限らず、他の許認可・資格・法務領域の専門知識を同様の構造（診断ロジック＋UI＋営業資料）でプロダクト化できるか。

2. **ロジック保護のルール（runtime_rules）はどこまで一般化できるか**
   AIエージェントがコードを誤改変するリスクへの対策は、この案件固有か、他のAI活用プロジェクトにも共通するノウハウとして整理できるか。

3. **docs/ による営業資料化は、営業担当者不在でもスケールするか**
   1枚資料・スライド構成・ヒアリングシートまでテンプレート化されていることで、営業経験の薄いメンバーでも面談に持ち込める状態になっているか、実際に試してみたい。

4. **PoCから本格導入への移行基準をどう設計するか**
   `docs/poc/tokutei_navi_poc_design.md` の成功条件は初期診断としては妥当だが、本格導入判断のための定量的な基準（利用継続率、企業側の満足度等）をどう追加すべきか。

5. **免責・法務面のドキュメント（docs/legal/）を誰がメンテナンスするか**
   制度改正等があった場合に、`docs/legal/disclaimer.md` や診断ロジックの前提をどう継続的に見直す体制にするか。

6. **複数プロダクト（tokutei-navi-01 / PreCheck / tokutei-integrity-01）のリポジトリ間連携をどう管理するか**
   現状はリンクURLでの接続のみだが、今後データ連携や共通の営業資料フォーマットを持たせる必要があるか。
